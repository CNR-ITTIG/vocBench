Getting started

About the Workbench

The AGROVOC workbench is the tool used to manage concepts, terms, schemes, and relationships between them in AGROVOC. AGROVOC is a multilingual structured thesaurus of all subject fields in Agriculture, Forestry, Fisheries, Food security and related domains.

Basic functionality

The Workbench is managed collaboratively among users with different roles. To understand how the Workbench functions, it is important to understand the distinction between concepts and terms.

Depending on your user role and privileges, you will be able to perform different actions in the Workbench.

Throughout the Workbench, you will see three types of square icons:
  1. Square icons with [-]. Use this icon to delete the information it is associated with.

  2. Square icons with [+]. Use this icon to add to the information you are viewing.

  3. Square icons with a small pencil. Use this icon to edit information.

A legend also shows you the status and type of the content you are viewing.

Content

The following modules make up the content in the Workbench:
  1. Search Module: use this module to search concepts, terms, relationships and other information.

  2. Concepts module: use this module to browse for concepts and see associated terms and information. Also use this module to change, add or delete relationships between concepts and terms.

  3. Relationships: use to manage the relationships in the ontology. To work with relationships between concepts and terms, use the Concepts Module. To work with relationships within schemes, use the Schemes Module. Full use of this module is available only for publishers and administrators.

  4. Schemes: use this module to view and manage classifications and schemes and the relationships within them.

  5. Validation: this module is for administrators, publishers, and validators to validate suggestions and changes made by other users.

  6. Consistency: this module is used to perform quality control.

  7. Export: this module is used to export data chosen by the user.

  8. Statistics: this module is for administrators, publishers, and validators to view information on the amount and type of use on the Workbench.

  9. Users: this module is for administrators, publishers, and validators to manage users.

  10. Groups: allows administrators and other authorized users to manage groups.

  11. Preferences: this module allows users to set viewing and language preferences.


Language

By default, all the terms associated with a concept appear when viewing a concept. You may wish to change the language settings during your use of the Workbench.

Language settings

If you wish to set languages for your current session only, use the "Customize language display" button at the top right of every page. If you wish to set preferences for future sessions, use the Preferences Module to set your preferred languages.

Language of interface

The interface - the information you see that does not include concepts, terms and relationships - is currently only available in English. More languages will be available soon and they will be available from the top of every page.

Language of the data

You can choose which languages you wish to see concepts displayed in. The full representation of a concept is all of the terms used to represent it, in all languages but reducing the number of languages displayed can make browsing concepts easier.

You can choose in which language(s) you would like to see concepts displayed for the current session by selecting the "Customize language display" button at the top right of every page.

Adding languages to the Workbench

If you have permission to add languages to the Workbench, you can do this using the "Customize language display" button at the top right of every page.

  • A blue square with a T denotes a term


Status of the data

The legend for the status of concepts and terms and the relationships between them can be seen at the bottom of the Concepts, Schemes and Relationships Modules.

The following statuses are included in the legend:
Proposed (guest): a non-logged in, anonymous guest user has proposed a term that will be assessed by an authorized and registered validator.
Proposed: a registered user has proposed a change.
Published: a validated change has been approved and confirmed by a publisher and and is now a published part of the ontology.
Validated: a validator has approved the change. Once a publisher reviews it, it will become published.
Revised (guest): a non-logged in, anonymous guest user has changed content that will be assessed by an authorized and registered validator.
Revised: a registered user has made a revision.
Proposed deprecated: a registered user has proposed to remove a concept or term.
Deprecated: a term or concept has been approved as being outdated.

User roles and privileges

Note: Please be aware that at the moment, above described user rights and permissions are only partly implemented.

Users with different roles maintain the data in the workbench.

All users are allowed to create, edit and delete concepts, terms and relationships. Their roles will determine how these actions appear to administrators:

  • The creation of a new term or concept will enter the system with the status 'proposed' or 'guest-proposed', depending on if you are a registered user or logged in on a guest account.

  • If a non-logged-in user deletes a concept, term or relationship, it will appear as 'proposed deprecated'.

All actions will be reassessed and validated by authorized registered validators and publishers.

The following are identified user roles. Click on the links to read more details about them.
  1. Guests (non logged in users) can make suggestions for new terms and other suggestions about concepts and relationships, such as proposing to delete or change.

  2. Term editors (terminologists) can manage the entire terminology base in their own language(s).

  3. Ontology editors can manage concepts and relationships.

  4. Validators can check (approve or reject) the changes done by the users in numbers 1-3 in this list. They can also approve new registered users.

  5. Publishers are responsible for publishing the validated entities. Only published entities will appear in exports that are used for indexing and other purposes.

  6. Super users (administrators) have full access to the system.


The implementation of different user roles is ongoing. The goal is to have a workflow with the status of data changes based on user actions.

Guests (Non-logged in users)

Guests (non-logged in users) can perform the following actions:
  • Navigate or search the system.

  • Submit suggested terms in any language.

  • Other suggestions about concepts and relationships, such as proposing to delete or change.


All suggestions will be identified as “guest proposed” and will be assessed by an authorized and registered validator.

Most guest proposals will be visible as proposals made by guests in the system. However, proposed deletions or edits of existing terms and concepts will only show upon successful validation of a validator.

All suggestions made by these users should be associated with their IP address.

Term editors

Term editors include terminologists and thesaurus editors. The following describes their roles and privileges:
  • Full permissions on terms. which they own (i.e. created themselves).

  • Any changes in terms which they own, such as deleting or editing, will be published without additional validation through a validator.

  • Term editors who are the owner/creators of a term can delete it completely.

  • Any other changes term editors make in the system will be proposals that need to be validated.

  • Permission to create terms in his or her designated language(s) without additional validation.

  • Modifications on concepts and terms which they are not the owner of will be set as "proposed”.

  • Every term editor has set one or more working languages for themselves and can only add/edit/delete terms of that language. Edit access for other languages is restricted.

  • Editors can provide suggestions to more experienced users (e.g. concepts modifications that will be processed by ontology editors).


Ontology editors

Ontology Editors include more experienced terminologists and thesaurus editors. The following describes their role:
  • They are terminologists/thesaurus editors familiar with ontologies and the distinction of concepts and terms.

  • They manage terms (of specific languages, i.e. the ontology editor's registered languages) and concepts.

  • They have permission to create and publish a new concept without further validation.

  • Only edits on concepts of which the ontology editor is not the owner will enter the system as proposed and will have to be validated.

  • If an ontology editor deletes a concept, only the terms of the corresponding languages will be set to “proposed deprecated” and the concept status will be set to “proposed deprecated”.

  • Ontology editors can completely delete a concept if they are the owner/creator of a concept and have privileges in all the languages of the terms of the concept.


Validator

The following describes validators' roles and privileges:
  • Validators are experts that can check and validate in their respective languages the work done by term editors, ontology editors and guests.

  • They validate all proposals and guest proposals and have the power to reject proposals, i.e. remove them again from the model or reset them to their previous state (in case of edits).

  • If a change is validated by validator, its status then needs to be validated by publisher to get a published status.

Publishers

The following describes publishers' roles and privileges:
  • Permission on terms and concepts.

  • Can validate work by editors and confirm validator work.

  • They ultimately approve a validated entry by a validator and change its status to "published".

  • They can still reject validated entries and thus remove them again from the model (in case of new proposals) or reset them to their previous status (in case of an edit or proposal to delete).

  • Access to all languages.


Administrators

The following describes the roles and privileges of administrators:
  • Permissions that are included in all user roles.

  • Permission to change manually the status of a concept or a term (for all other users a status will be set by the system automatically based on the action)

  • Can manage users and groups.

  • Can provide statistics.

  • Access to all languages.
